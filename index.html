    Paper

    Abstract. After nearly five years and 34 draft versions, standardization of the new connection oriented transport protocol QUIC is finally on the horizon. We present the first large scale analysis of Internet Engineering Task Force (IETF) QUIC deployments and supported versions on the Internet based on a newly implemented IPv4 and IPv6 ZMap module using the QUIC version negotiation to uncover QUIC deployments and used versions. We infer additional QUIC deployments from application layer protocol negotiation values indicating HTTP/3 support, collected through DNS scans of the newly drafted HTTPS DNS resource record and HTTP Alternative Service Header values from successful HTTP handshakes.
    In total, we find more than 2.3 M IPv4 and 300 k IPv6 addresses supporting QUIC even though deployment is strongly dominated by large providers. We show that a transition to newer versions based on the IETF draft can be seen over time and that multiple deployments, including all of Cloudflare, already activate QUIC “Version 1”.
    Using our newly implemented stateful QUIC scanner (QScanner) we are able to successfully scan 26 M targets. However, we show that up to 8 % of scans result in version mismatches especially with targets from Google, even though servers indicated support in the version negotiation.
    Overall, we identify 44 widely varying transport parameter configurations, e.g., with differences in the order of magnitudes for performance relevant parameters. Combining these configurations with HTTP Server header values and associated domains reveals two large edge deployments from Face book and Google. Thus, while found QUIC deployments are located in 4667 autonomous systems, numerous of these are again operated by providers. In our experience, IETF QUIC already sees an advanced deployment status even before the final standardization mainly driven by large providers.
